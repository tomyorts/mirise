# 04 Security Privacy Medical

## 基本方針

医療現場のインカムは、単なる社内チャットではありません。
会話に患者名、症状、治療内容、予約状況、会計情報が含まれる可能性があります。
したがって、初期設計からプライバシー保護を前提にします。

## 初期版で保存しないもの

- 音声録音
- 文字起こし
- AI要約
- 患者名
- 診療内容
- 予約詳細
- 会計内容

## 保存してよいもの

- スタッフID
- 所属院
- 参加ルーム
- 接続開始・終了時刻
- エラー種別
- 端末種別
- 通信品質メタデータ

通信品質メタデータにも、会話内容は含めません。

## 認証方針

MVP:

- スタッフ名入力のみ
- 院内PoC限定
- 外部公開しない

本番:

- スタッフアカウント
- 施設別権限
- JWTトークン短期発行
- 退職者即時無効化
- 端末紛失時の無効化
- 管理者権限の分離

## 通信方針

MVP:

- 院内LAN限定
- HTTPSなしでも可。ただしPoCのみ

本番:

- HTTPS/WSS必須
- TURN/TLS対応
- APIシークレットをクライアントに置かない
- 管理画面はIP制限または二要素認証

## 会話ルール

現場ルールとして以下を明文化します。

- 患者フルネームを言わない
- 診断名や治療内容を不用意に言わない
- 必要な場合は院内コードやチェア番号で伝える
- 患者前で不適切な内容を話さない
- 緊急時を除き、全体ルームを濫用しない

例:

悪い例:

```text
山田太郎さん、右下6番の抜歯準備お願いします
```

良い例:

```text
3番チェア、外科セットお願いします
```

## 録音について

初期版では録音しません。

録音は、教育・品質管理に使える一方で、以下のリスクが大きいです。

- 患者情報の保存
- スタッフ監視感
- 同意取得の問題
- 保管期間・削除・開示対応
- 漏えい時の影響

録音を検討する場合は、目的、同意、保存期間、アクセス権、削除手順、法務確認を先に決めます。

## AI文字起こしについて

初期版では入れません。

理由:

- 患者情報がテキストとして残る
- 誤認識が業務混乱を招く
- 外部AI利用時の委託・責任分界が必要
- スタッフの心理的抵抗が強くなりうる

AI活用は、音声内容ではなく通信品質分析やノイズ抑制から始める方が安全です。

## 医療情報に関する注意

診療・調剤に関する情報や、医療従事者が知り得た患者の身体状況、病状、治療状況等は、要配慮個人情報として扱うべき情報です。
そのため、音声インカムは患者情報を保存しない前提で設計します。

## リスク一覧

| リスク | 影響 | 対策 |
|---|---|---|
| 患者情報が音声に乗る | プライバシー侵害 | 会話ルール、録音なし |
| 端末紛失 | 不正参加 | 端末無効化、短期トークン |
| 退職者がアクセス | 内部不正 | アカウント即時停止 |
| Wi-Fi盗聴 | 情報漏えい | 暗号化、SSID分離 |
| 外部サーバー障害 | 通話不可 | 院内フォールバック |
| 誤って全体発話 | 現場混乱 | UI確認、全体ルーム制御 |
